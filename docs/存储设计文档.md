> 版本：v1.0
> 状态：设计阶段

## 一、概述

本文档定义 DeviceStudio 的数据存储策略，包括日志存储、历史数据存储、配置文件组织和数据索引设计。

### 1.1 存储目标

| 目标 | 说明 |
|-----|------|
| **持久化** | 设备配置、用户设置、协议定义等需要持久保存 |
| **追溯性** | 通信日志、操作记录可追溯 |
| **高效性** | 历史数据查询快速 |
| **可移植** | 数据格式跨平台兼容 |

### 1.2 存储架构

```
┌─────────────────────────────────────────────────────────────┐
│                      应用程序                                │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐        │
│  │ 配置管理 │  │ 日志系统 │  │ 数据中心 │  │ 插件管理 │        │
│  └────┬────┘  └────┬────┘  └────┬────┘  └────┬────┘        │
├───────┴───────────┴───────────┴───────────┴─────────────────┤
│                      存储抽象层                              │
├─────────────────────────────────────────────────────────────┤
│    JSON      │     SQLite     │    Binary    │    CSV      │
│  配置/协议   │   历史数据     │   日志文件   │   数据导出   │
└──────────────┴────────────────┴──────────────┴──────────────┘
```

## 四、日志存储设计

### 4.1 日志文件格式

日志采用纯文本格式，按日期分割文件。

**文件命名规则：**
```
{type}_{date}.log

示例:
app_2025-02-14.log
communication_2025-02-14.log
error_2025-02-14.log
```

### 4.2 日志格式定义

```
[时间戳] [级别] [模块] 消息内容 {附加数据}

示例:
[2025-02-14 10:23:45.123] [INFO] [SerialDevice] 连接成功 {port: "COM3", baud: 115200}
[2025-02-14 10:23:46.456] [DEBUG] [ProtocolEngine] 解析数据帧 {frame: "AA5501...", type: "SensorData"}
[2025-02-14 10:23:47.789] [ERROR] [DeviceManager] 连接超时 {device: "dev001", timeout_ms: 5000}
```

### 4.3 日志级别

| 级别 | 标识 | 说明 |
|-----|------|------|
| TRACE | TRACE | 详细跟踪信息 |
| DEBUG | DEBUG | 调试信息 |
| INFO | INFO | 一般信息 |
| WARNING | WARN | 警告信息 |
| ERROR | ERROR | 错误信息 |
| CRITICAL | CRIT | 严重错误 |

### 4.4 日志文件轮转策略

```
日志文件轮转规则:

1. 按日期分割
   - 每天创建新文件
   - 文件名: app_YYYY-MM-DD.log

2. 按大小分割
   - 单文件最大 10MB（可配置）
   - 超过大小后创建新文件: app_YYYY-MM-DD_001.log

3. 保留策略
   - 默认保留 30 天
   - 超过天数自动删除
   - 可配置最大文件数
```

### 4.5 通信日志格式

通信日志记录所有收发数据，用于问题追溯和数据分析。

```json
{
    "timestamp": "2025-02-14T10:23:45.123Z",
    "device_id": "dev001",
    "direction": "rx",
    "data_hex": "AA5501640000",
    "data_ascii": "....d..",
    "length": 6,
    "parsed": {
        "frame_type": "SensorData",
        "fields": {
            "device_id": 1,
            "temperature": 25.5,
            "status": "normal"
        }
    }
}
```

或文本格式：

```
[2025-02-14 10:23:45.123] RX dev001 [6 bytes] AA 55 01 64 00 00
└─ 解析: SensorData {device_id: 1, temperature: 25.5°C, status: normal}

[2025-02-14 10:23:46.456] TX dev001 [4 bytes] AA 55 02 01
└─ 指令: QueryStatus
```

## 五、历史数据存储

### 5.1 存储方案选择

| 方案 | 优点 | 缺点 | 适用场景 |
|-----|------|------|---------|
| SQLite | 轻量、单文件、查询灵活 | 写入性能一般 | 中小规模数据 |
| CSV | 简单、可读、易导出 | 查询不便 | 数据导出 |
| 二进制 | 高效、紧凑 | 不可读 | 高频数据采集 |
| 时序数据库 | 专业、高效 | 依赖重 | 大规模生产环境 |

**DeviceStudio 采用 SQLite + CSV 导出方案。**

### 5.2 SQLite 数据库结构

#### 5.2.1 数据库文件命名

```
data/history/{date}.db

示例:
data/history/2025-02-14.db
data/history/2026-03-15.db
```

#### 5.2.2 表结构设计

**数据通道表 (channels):**
```sql
CREATE TABLE channels (
    id TEXT PRIMARY KEY,        -- 通道ID
    name TEXT NOT NULL,         -- 显示名称
    device_id TEXT,             -- 关联设备ID
    unit TEXT,                  -- 单位
    description TEXT,           -- 描述
    min_value REAL,             -- 最小值
    max_value REAL,             -- 最大值
    color TEXT,                 -- 显示颜色
    created_at INTEGER,         -- 创建时间戳
    metadata TEXT               -- 元数据 (JSON)
);
```

**数据点表 (data_points):**
```sql
CREATE TABLE data_points (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    channel_id TEXT NOT NULL,   -- 通道ID
    timestamp INTEGER NOT NULL, -- 时间戳 (毫秒)
    value REAL NOT NULL,        -- 数值
    metadata TEXT,              -- 元数据 (JSON)
    FOREIGN KEY (channel_id) REFERENCES channels(id)
);

-- 索引
CREATE INDEX idx_data_points_channel ON data_points(channel_id);
CREATE INDEX idx_data_points_timestamp ON data_points(timestamp);
CREATE INDEX idx_data_points_channel_time ON data_points(channel_id, timestamp);
```

**设备会话表 (device_sessions):**
```sql
CREATE TABLE device_sessions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    device_id TEXT NOT NULL,    -- 设备ID
    start_time INTEGER,         -- 连接时间
    end_time INTEGER,           -- 断开时间
    bytes_received INTEGER,     -- 接收字节数
    bytes_sent INTEGER,         -- 发送字节数
    frames_received INTEGER,    -- 接收帧数
    errors INTEGER,             -- 错误数
    status TEXT                 -- 结束状态
);
```

**解析帧记录表 (parsed_frames):**
```sql
CREATE TABLE parsed_frames (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp INTEGER NOT NULL, -- 时间戳
    device_id TEXT,             -- 设备ID
    frame_type TEXT,            -- 帧类型
    raw_data BLOB,              -- 原始数据
    parsed_data TEXT,           -- 解析结果 (JSON)
    is_valid INTEGER            -- 是否有效
);

CREATE INDEX idx_parsed_frames_time ON parsed_frames(timestamp);
CREATE INDEX idx_parsed_frames_device ON parsed_frames(device_id);
```

## 九、存储性能优化

### 9.1 批量写入

```cpp
// 批量写入数据点（减少数据库事务）
void DataCenter::appendDataBatch(const QList<DataPoint>& points) {
    db.transaction();
    QSqlQuery query;
    query.prepare("INSERT INTO data_points (channel_id, timestamp, value) VALUES (?, ?, ?)");
    for (const auto& point : points) {
        query.addBindValue(point.channelId);
        query.addBindValue(point.timestamp);
        query.addBindValue(point.value);
        query.exec();
    }
    db.commit();
}
```

### 9.2 索引优化

```sql
-- 复合索引优化查询
CREATE INDEX idx_data_points_channel_time 
ON data_points(channel_id, timestamp);

-- 使用覆盖索引
CREATE INDEX idx_data_points_cover 
ON data_points(channel_id, timestamp, value);
```

### 9.3 内存缓冲

```
写入流程:

数据点 → 内存缓冲区 → 批量写入 → SQLite
          │
          └─ 缓冲区满/超时触发写入

配置:
{
    "buffer_size": 1000,      // 缓冲区大小
    "flush_interval_ms": 1000 // 刷新间隔
}
```

## 十、存储空间估算

### 10.1 单设备数据量估算

| 项目 | 估算值 | 说明 |
|-----|-------|------|
| 每秒数据点 | 10 | 假设 10 个通道，每通道 1Hz |
| 每小时数据点 | 36,000 | 10 × 3600 |
| 每天数据点 | 864,000 | 10 × 86400 |
| 单数据点大小 | ~30 bytes | 含索引 |
| 每天数据量 | ~26 MB | 864,000 × 30 |

### 10.2 存储空间规划

```
预估存储需求（单设备，10通道，1Hz采样）:

1天:    ~26 MB
7天:    ~182 MB
30天:   ~780 MB
90天:   ~2.3 GB

加上日志和备份，建议预留 5GB/设备/季度
```

